## Extension Boot Strategy

### How Extensions Are Loaded

All extensions (except root-level) are loaded via the justfile using a **single-entry dynamic loader**:

```bash
just ext-full-stack
  → PI_STACK="agent-team,damage-control,theme-cycler"
  → pi -e .pi/extensions/util/pi-loader.ts
  → pi-loader.ts resolves, categorizes, and initializes each extension
```

### Extension Layers

| Layer | Directory | Files | Role |
|-------|-----------|-------|------|
| Entry | `ui/` | 6 | Primary entry-point extensions |
| Middle | `util/` | 12 | Utilities, memory, loaders |
| Base | `src/ui/` | 10 | Shared UI components, themes |

### Path Resolution Order

When `pi-loader.ts` looks for an extension by name, it searches:

```
1. extensions/<name>.ts
2. extensions/<name>.js
3. .pi/extensions/<name>.ts
4. .pi/extensions/ui/<name>.ts
5. .pi/extensions/util/<name>.ts
6. .pi/extensions/src/ui/<name>.ts
```

## Key Files Explained

### `justfile` (root)

The single entry point for running Pi with extensions. Defines ~25 targets that set `PI_STACK` and load `pi-loader.ts`. Never pass multiple `-e` flags directly — use the justfile.

### `.pi/damage-control-rules.yaml`

Safety configuration with four rule categories:

| Section | Purpose | Entries |
|---------|---------|---------|
| `bashToolPatterns` | Dangerous command patterns to block or flag | 60+ patterns (rm, git, aws, gcloud, SQL, etc.) |
| `readOnlyPaths` | Paths that agents can read but not modify | 50+ paths (.env, SSH keys, cloud creds, lock files, build dirs) |
| `noDeletePaths` | Paths that cannot be deleted | 20+ paths (.git, LICENSE, README, CI configs, Dockerfile) |
| `pathOverrides` | Highest-priority per-path permission rules | 7 active overrides |

Patterns marked with `ask: true` prompt the user before executing. Patterns without `ask` are blocked entirely.

### `.pi/teams.yaml`

Legacy team definitions — simple lists of agent names. **Superseded by** `.pi/agents/teams.yaml` which has more detailed team configurations (13 teams vs 6 here). Kept for backward compatibility.

| Team | Members |
|------|---------|
| `main` | scout, planner, developer, reviewer, documenter, red-team, session-manager |
| `plan-build` | planner, developer, reviewer, session-manager |
| `info` | scout, documenter, reviewer, session-manager |
| `frontend` | planner, frontend-coder, scout, reviewer, bowser, session-manager |
| `pi-pi` | ext-expert, theme-expert, skill-expert, config-expert, tui-expert, prompt-expert, agent-expert, session-expert |
| `Extra` | developer, reviewer, scout, dispatcher |
